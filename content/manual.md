# 第1章 全体フローの理解

## 1-1 申込から入居までの全体マップ

UCJapanの入居案件は、入居者の本申込から入居完了まで、複数の工程を経て進行します。各工程はAsanaのプロジェクトに対応しており、案件は番号順にプロジェクトを移動していきます。

### 案件進行の流れ

| 工程 | Asanaプロジェクト | 主な状態 |
|---|---|---|
| 1 | 1.本物件 | 物件マスター情報 |
| 2 | 2.申込 | 本申込完了から審査中、審査結果確定まで |
| 3 | 3.送金前 | 起算日調整、初期費用請求書受領、入金待ち、管理会社送金前まで |
| 4 | 4.送金済み | 契約書類受領、重説書類作成、IT重説実施、入居前まで |
| 5 | 5.入居済み | 入居完了後 |

各プロジェクトの詳細は **第2章「Asanaによる進捗管理」** で扱います。

### 工程ごとの主要イベント

申込から入居までの流れを、案件の動きの観点でまとめると以下のようになります。

**段階1: 申込・審査**
- 入居者が物件を選定し、本申込
- 管理会社の審査
- 審査結果の確定(通過 / 落ち / 2番手)

**段階2: 起算日と費用**
- 起算日の確定(管理会社・入居者と調整)
- 管理会社から初期費用の請求書を受領
- 入居者からUCへの入金
- UCから管理会社への送金

**段階3: 契約手続き**
- 管理会社から契約書類(雛形)を受領
- 重説書類の作成(別チーム)
- 重説書類の事前確認
- IT重説の実施
- 入居者の電子契約完了

**段階4: 入居**
- 鍵渡し、入居
- 入居後の管理対応(必要に応じて入居者連絡先を管理会社へ伝達など)

### 宅建士が動き始めるタイミング

案件は **韓国チームによる Asana タスク作成 → 日本チームの別担当による本申込実施 → 宅建士への担当割り振り** という流れで宅建士の手元に届きます。

宅建士が実質的に業務を開始するのは、**Asanaのタスクが「2.申込」プロジェクトの「申込中」セクションに到達した時点** からです。それ以前(「申込準備中」セクション)は、宅建士の対応対象ではありません。

### ここまでのポイント

- 案件はAsanaのプロジェクトを番号順に移動する(2.申込 → 3.送金前 → 4.送金済み → 5.入居済み)
- 大きな段階として「申込・審査」「起算日と費用」「契約手続き」「入居」の4段階
- 業務開始タイミングは「2.申込/申込中」セクション到達時

## 1-2 関係者と役割分担

UCJapanの入居案件には、社内外の多数の関係者が関わります。宅建士として円滑に業務を進めるには、誰がどの役割を担っているかを理解しておくことが重要です。

### 社外の関係者

| 関係者 | 役割 | 宅建士との接点 |
|---|---|---|
| **入居者** | 物件を借りる人(主に韓国からの渡日者) | 直接コミュニケーションは取らない(韓国チーム経由) |
| **管理会社** | 物件の管理を行う会社(スカイコート、アムスエステート、ジャパンレジアスなど多数) | 管理会社チャットでやり取り(管理会社ごとに使用ツールが異なる) |
| **保証会社** | 入居者の保証を引き受ける会社(環境ステーション、フィニックス、アンビションDXホールディングス、エイマックス、ピカソ、リブマックスなど) | 必要に応じて書類提出など |

物件情報の入手元は、UCJapanの **自社運営の物件紹介サイト**(OPEN SEET 等)です。

### 社内の関係者

| チーム / 担当 | 役割 | 宅建士との接点 |
|---|---|---|
| **韓国チーム** | 入居者ヒアリング、申込書類作成、入居者連絡先での直接対応 | Slack「質問_韓国チームへ」「質問_日本チームへ」で連携 |
| **日本チーム別担当(本申込)** | 仮申込から本申込への切り替え、宅建士への担当割り振り | Asanaで担当が割り振られる起点 |
| **経理担当** | 入金確認、管理会社送金、初期費用の請求書フロー進行 | Asanaのフィールド更新で連携 |
| **重説作成担当** | 重説書類の作成(契約書類雛形を元に作成) | 「契約書」フィールドを「作成依頼済」にした後に作業開始 |
| **家電レンタル斡旋担当** | 家電レンタル関連の調整 | 宅建士の業務範囲外 |
| **lifeline調整担当** | 電気・水道・ガスの調整 | 宅建士の業務範囲外 |

### 入居者とのコミュニケーション経路

UCJapanでは、入居者との直接コミュニケーションは **韓国チームのみが行う** 体制です。宅建士が入居者と直接やり取りすることは基本的にありません。

入居者に対して何かを伝えたい / 確認したいときは、Slackの「質問_韓国チームへ」チャンネル経由で韓国チームに依頼し、韓国チームから入居者へ伝達する流れになります。

韓国チームから日本チームへの連絡は、Slackの「質問_日本チームへ」チャンネルで受け取ります。

### 管理会社とのコミュニケーション経路

管理会社とのやり取りは、**宅建士が窓口** となります。使用するチャットツールは管理会社ごとに異なります。

### ここまでのポイント

- 入居者とは **直接やり取りしない**(韓国チーム経由)
- 管理会社とは **宅建士が直接窓口**(チャットツールは管理会社ごとに異なる)
- 経理、重説作成、家電、lifeline は社内の別担当
- 案件は韓国チーム → 日本チーム別担当 → 宅建士 の順に手元に届く

## 1-3 宅建士の業務範囲

このマニュアルで扱う宅建士の業務範囲は、社内の他部門との分業を前提に明確に定義されています。「やること」と「やらないこと」を最初に押さえることで、業務上の迷いを減らせます。

### 宅建士がやること

| カテゴリ | 業務内容 |
|---|---|
| **管理会社とのやり取り** | 申込から入居までの管理会社チャット対応全般(審査中の追加書類対応、起算日調整、契約書類の受領、入居者連絡先の伝達など) |
| **起算日・入居日の調整** | 管理会社との起算日確定、入居者との起算日希望日のすり合わせ(韓国チーム経由) |
| **初期費用請求書の処理** | 管理会社から受領した請求書をGoogleドライブ「【申込チーム】請求書格納フォルダ」直下にアップロード、Asana「初期費用」フィールドを「作成依頼済」に変更 |
| **契約書類の処理** | 管理会社から受領した契約書類雛形をGoogleドライブ「【申込チーム】重要事項作成フォルダ」直下にアップロード、Asana「契約書」フィールドを「雛形アップ済」→「作成依頼済」と段階更新 |
| **重説書類の事前確認** | 別チームが作成した重説書類の内容チェック(第4章) |
| **IT重説の実施** | 入居者へのIT重説実施(第5章) |
| **Asanaの担当フィールド更新** | 起算日状況、審査結果、初期費用、契約書フィールドの宅建士担当分の更新 |
| **在留カードの管理会社送付** | 管理会社の要求があった場合のみ、韓国チーム経由で取得して送付 |
| **入居者連絡先の伝達** | 管理会社の要求があった場合のみ、Asanaの「携帯」フィールドの番号を伝達 |

### 宅建士がやらないこと

以下は社内の別担当または別部門の業務です。Asanaタスク上で関連フィールドが見えますが、宅建士が編集する必要はありません。

| カテゴリ | 担当 |
|---|---|
| **入居者との直接コミュニケーション** | 韓国チーム |
| **本申込の実施 / 担当割り振り** | 日本チーム別担当 |
| **入金確認(GMOあおぞらネット銀行等の入金チェック)** | 経理担当 |
| **管理会社への送金実行** | 経理担当 |
| **入居者向け請求書の作成・送付** | 経理担当 |
| **重説書類の作成** | 重説作成担当(別の社内チーム) |
| **IT重説後の「読み実施済み」フィールド更新** | 別担当 |
| **家電レンタル斡旋(家電フィールド全般)** | 家電レンタル斡旋担当 |
| **lifeline調整(電気・水道・ガス)** | lifeline調整担当 |
| **入居者からの在留カード取得** | 韓国チーム |
| **顧客紹介プログラム(n번째추천예정 等)** | 韓国チーム |
| **鍵渡し(鍵渡しフィールド)** | 別担当 |

### Asanaフィールドの「触る / 触らない」早見表

各プロジェクトで宅建士が触るフィールドをまとめると以下のようになります。

**2.申込で触るフィールド**
- 起算日状況(決定 / 相談 / 未定)
- 審査結果(通過 / 落ち / 2番手)

**3.送金前で触るフィールド**
- 初期費用 → 作成依頼済
- 起算日(確定日付の入力)

**4.送金済みで触るフィールド**
- 契約書 → 雛形アップ済 → 作成依頼済(2段階)

**5.入居済みで触るフィールド**
- なし(管理会社の要請に応じて携帯番号を読み取って伝えるのみ)

それ以外のフィールドは、表示されていても触りません。

### ここまでのポイント

- 主軸は「管理会社との窓口業務」と「IT重説実施」
- Asanaのフィールド更新は担当範囲が明確に決まっている
- 「やらないこと」を理解することで、他部門の業務を侵食しない

## 1-4 使用ツールと役割

業務で使用する主なツールは以下の通りです。それぞれの役割を理解した上で、目的に応じて使い分けます。

### ツール一覧

| ツール | 主な役割 | 詳細を扱う章 |
|---|---|---|
| **Asana** | 案件進捗管理、ステータスの可視化、自動化ルールによる工程移動 | 第2章 |
| **Slack** | 社内の連絡・依頼・情報共有(4チャンネル) | 第3章 |
| **管理会社チャット** | 管理会社との直接のやり取り(管理会社ごとにツールが異なる) | 第2章 / 第3章で実務的に登場 |
| **Googleドライブ** | 請求書・契約書類の保管(社内共有用) | 第2章(2-6、2-7)で実務的に登場 |
| **Google Meet** | IT重説の実施 | 第5章 |

### ツール間の連動関係

**Asana ⇔ Slack**
- Asanaの自動化ルールによって、特定のイベント発生時にSlackの「asana_日本チーム」チャンネルへ自動通知が流れる
- 通知例: 起算日決定、読み合わせ決定、担当者振り当て、申込キャンセル

**Slack ⇔ 管理会社チャット**
- 管理会社から受け取った情報を、社内のSlackで関係者に共有
- Asanaタスクの説明欄に管理会社チャットのURLを貼り付けて、社内メンバーが直接アクセスできるようにする

**Asana ⇔ Googleドライブ**
- Asana上で管理会社から受領した書類(請求書・契約書類)を、Googleドライブの指定フォルダにアップロード
- アップロード後にAsanaのフィールドを更新することで、次工程の担当者が動き始める

### Googleドライブのフォルダ構造

宅建士がアップロード先として使う主なフォルダは2つです。

| 用途 | フォルダ名 | アップロード先 |
|---|---|---|
| **初期費用請求書(管理会社→UC)** | 【申込チーム】請求書格納フォルダ | フォルダ直下 |
| **契約書類雛形(管理会社→UC)** | 【申込チーム】重要事項作成フォルダ | フォルダ直下 |

いずれもサブフォルダには入れず、親フォルダ直下に置きます。ファイル名は「**案件番号+入居者名+物件名.pdf**」の形式に変更します(Asanaタスク名と同じ命名規則)。

### 業務での使い分けの基本パターン

| やりたいこと | 使うツール |
|---|---|
| 自分の担当案件の状況を確認したい | Asana(2.申込 〜 5.入居済みのいずれかのプロジェクトを開く) |
| 起算日が決定したかチェックしたい | Slack「asana_日本チーム」(自動通知) |
| 入居者に何かを確認・伝達したい | Slack「質問_韓国チームへ」(韓国チーム経由) |
| 管理会社に問い合わせたい | 管理会社チャット(管理会社ごとに異なる) |
| 社内の契約担当に確認したい | Slack「契約チーム」 |
| 韓国チームから来た質問に答えたい | Slack「質問_日本チームへ」 |
| 管理会社からもらった請求書を保管したい | Googleドライブ「【申込チーム】請求書格納フォルダ」 |
| 管理会社からもらった契約書類を保管したい | Googleドライブ「【申込チーム】重要事項作成フォルダ」 |
| IT重説を実施したい | Google Meet |

### ここまでのポイント

- 主要ツールは Asana / Slack / 管理会社チャット / Googleドライブ / Google Meet
- ツール同士は連動しており、特に Asana ⇔ Slack の自動通知は重要
- 書類の保管先 Googleドライブのフォルダは用途別に2つ
- 業務シーンに応じてツールを使い分ける

# 第2章 Asanaによる進捗管理

## 2-1 Asanaとは何か / なぜ使うのか

UCJapanでは、入居申込から入居完了までの全案件を **Asana(アサナ)** というプロジェクト管理ツールで管理しています。

### Asanaを使う理由

弊社の案件は、申込から入居まで以下のような複数のステップを踏みます。

- 入居者からの申込受付(韓国チーム)
- 管理会社とのやり取り(宅建士)
- 入居者からの入金確認(経理担当)
- 管理会社への送金(経理担当)
- IT重説の実施(宅建士)
- ライフラインの調整(別担当)
- 入居・鍵渡し
- 家電レンタル斡旋(別担当)

各ステップで担当者が異なり、また同時に複数の案件が並行して進んでいます。これを Slack やメールだけで管理すると、「今この案件はどの段階にあるのか」「次は誰が何をすべきか」が分からなくなります。

Asana では、1つの案件 = 1つの「タスク」として扱い、進行段階に応じて自動的に適切な「プロジェクト」に移動していきます。担当者は、自分が見るべきプロジェクトを開けば、対応が必要な案件が一覧できるようになっています。

### 業務を始める前に押さえておきたい3つのポイント

1. **タスクは韓国チームが作成します**
   入居者からの本申込が完了した時点で、韓国チームが Asana にタスクを作成します。ゼロから案件を立ち上げる必要はありません。

2. **タスクは自動的にプロジェクト間を移動します**
   特定のフィールドを更新したり、他部門の作業が完了したりすると、Asanaのルール機能によってタスクが次のプロジェクトへ自動で移動します。手動でドラッグして動かす必要はほとんどありません。

3. **見えるフィールドが多くても、触るのは一部だけです**
   タスクの中には「家電」「lifeline」「入金日」など多数のフィールドが並んでいますが、これらの多くは他部門の担当です。宅建士が更新するフィールドは限られています(後述)。

## 2-2 サイドバーの見方とプロジェクト一覧

Asanaにログインすると、画面左側に **サイドバー** が表示されます。ここから案件のプロジェクト一覧にアクセスします。

### プロジェクト一覧(番号順に進行)

サイドバーの「プロジェクト」セクションには、UCJapanで使用するプロジェクトが番号付きで並んでいます。番号は案件の進行順を示しています。

| 番号 | プロジェクト名 | 内容 |
|---|---|---|
| 1 | 1.本物件 | 物件のマスター情報 |
| 2 | **2.申込** | 入居者の本申込が完了した案件 |
| 3 | **3.送金前** | 管理会社への送金待ちの案件 |
| 4 | **4.送金済み** | 送金完了から入居までの間の案件 |
| 5 | **5.入居済み** | 入居が完了した案件 |
| 6 | 6.最新管理 | 案件横断の管理ボード(別担当) |
| 8 | 8.입주후데이터 | 入居後データ管理(別担当) |
| 9 | 9.DATA | データ管理(別担当) |
| 9 | 9.他社契約+保留 | 他社契約・保留案件 |
| 99 | 99.従業員用 | 従業員向け管理 |

### 宅建士が日常的に開くプロジェクト

太字で示した **2.申込 / 3.送金前 / 4.送金済み / 5.入居済み** の4つが、宅建士の主な作業場所です。これら以外のプロジェクトはサイドバーに表示されていても、別部門が管理しているため通常は触りません。

### 1つの案件は複数のプロジェクトに同時所属します

ここがAsanaの重要な仕組みです。1つのタスク(案件)は、進行段階に応じて「2.申込」「3.送金前」「4.送金済み」「5.入居済み」を順番に移っていきますが、それと並行して常に「6.最新管理」にも所属しています。

タスクの詳細画面を開くと、「プロジェクト」欄に **2つのプロジェクト名** が表示されます。これは正常な状態です。

例: あるタスクが「3.送金前」段階にあれば、
- プロジェクト1: 3.送金前(進行ステータス)
- プロジェクト2: 6.最新管理(横断管理)

の両方に所属しています。

### ここまでのポイント

- サイドバーの番号は案件の進行順
- 主な作業場所は **2.申込 / 3.送金前 / 4.送金済み / 5.入居済み** の4つ
- 1案件 = 1タスクで、進行に応じてプロジェクト間を移動する
- ただし「6.最新管理」には常に並行所属する

## 2-3 タスクの基本構造

Asanaにおける「タスク」は、UCJapanでは **1件の入居案件** に相当します。タスクの中には案件に関する情報が体系的に格納されています。

タスクをクリックすると詳細画面が開き、以下のような構造になっています。

### (1) タスク名

画面上部に大きく表示される、案件を識別する名前です。UCJapanでは以下の命名規則で韓国チームが作成します。

```
[案件番号][入居者名(カタカナ)]([物件名] [部屋番号])
```

例: `1723パクアヒョン(スカイコート中村橋第2 407)`

- 案件番号: 連番(韓国チームが採番)
- 入居者名: カタカナ表記
- 物件名: 建物名
- 部屋番号: 入居予定の部屋

担当する宅建士がタスク名を変更することは基本的にありません。

### (2) 担当者

タスク名の下にある「担当者」欄に表示されます。誰がそのタスクを進行する責任を持つかを示します。

UCJapanでは、宅建士が担当するタスクの場合、担当者欄には宅建士本人のアカウントが入ります。タスクを引き継いだ際には、自分のアカウントに変更します。

### (3) 期日

タスクの対応期限です。期日が過ぎると赤く表示されます。

### (4) プロジェクト

1つのタスクは複数のプロジェクトに所属します。「プロジェクト 2」と表示され、進行ステータスのプロジェクト(2.申込 / 3.送金前 など)と、横断管理のプロジェクト(6.最新管理)の2つが並びます。

それぞれのプロジェクト名の右側に、現在所属している **セクション名** が表示されます(例: 「2.申込 申込中」)。セクションについては次節で説明します。

### (5) カスタムフィールド

プロジェクトの中盤に、案件に関する詳細情報が項目別に並んでいます。これを **カスタムフィールド** と呼びます。

カスタムフィールドの数は20個以上あり、プロジェクトごとに表示される項目も変わります。例として「2.申込」段階で表示される主なフィールドは以下です。

| フィールド名 | 用途 |
|---|---|
| サイト | 申込書の管理会社送付ステータス |
| 物件紹介者 | 物件を見つけたサイト名 |
| 内見 | 内見ステータス |
| 管理会社 | 物件の管理会社名 |
| 申込日 | 申込日付 |
| 審査結果 | 管理会社の審査結果 |

すべてのフィールドを覚える必要はありません。各プロジェクトで触るフィールドは限られており、それは後の節(2-5以降)でプロジェクトごとに説明します。

### (6) 説明欄

タスクの下部にある自由記述エリアです。UCJapanでは韓国チームが入居者の希望条件などを韓国語で記入していることが多く、また外部リンク(Googleスプレッドシート、イタンジ申込受付くんなど)も貼られています。

宅建士が最初にこのタスクを担当する際、説明欄に **管理会社チャットのURL(イタンジなど)を貼り付ける作業** があります。それ以降は基本的に読むことが中心で、頻繁に編集する機会は多くありません。

### ここまでのポイント

- タスク = 1件の入居案件
- タスク名は韓国チームが作成、宅建士は変更しない
- 担当者欄に自分のアカウントを入れる
- プロジェクトは2つ並ぶ(進行ステータス + 6.最新管理)
- カスタムフィールドは多いが、触るのは一部のみ
- 説明欄は管理会社チャットURLの貼り付けを行う

## 2-4 セクションとプロジェクト間移動の仕組み

Asanaにおいて、タスクの位置は2つのレベルで管理されています。

- **プロジェクト**: 大きな進行段階(2.申込 / 3.送金前 / 4.送金済み / 5.入居済み)
- **セクション**: プロジェクト内の小区分

この2階層構造が、案件の進行をきめ細かく管理する仕組みです。

### セクションとは

プロジェクトを開くと、タスクが横一覧(リストビュー)または縦並び(ボードビュー)で表示されます。その中で、タスクは **セクション** という見出しでグループ化されています。

**例: 「2.申込」プロジェクトのセクション**

- 申込準備中
- 申込中

セクション名は、そのプロジェクト内でタスクが今どの状態にあるかをさらに細かく示しています。タスク詳細画面で「プロジェクト」欄を見ると、プロジェクト名の右側に現在のセクション名が表示されます。

例: `2.申込 申込中` ← 「2.申込」プロジェクト内の「申込中」セクションにいる

### プロジェクト間移動はどのように起きるか

ここが業務理解の鍵となる部分です。

**重要な前提:**
プロジェクト間の移動(2.申込 → 3.送金前 など)は、宅建士が手動でタスクをドラッグして動かすのではなく、**Asanaの自動化ルール** によって動かされます。

具体的な仕組みは複雑で、複数のフィールド値や他部門の作業完了状態などが組み合わさってトリガーされます。「次のプロジェクトに移そう」と意識する必要はありません。

**心構え:**
「自分の作業が完了したら、決められたフィールドを更新する。あとはAsanaが自動でタスクを次の場所に動かしてくれる」

これだけ意識すれば十分です。

**タスクが思った場所に移動していない場合:**
最もよくある原因は **フィールドの更新順序を間違えて、自動化ルールのトリガー条件を満たさなかった** ケースです。各プロジェクトでフィールドを更新する順番には意味があるので、後の節で説明する手順を守ってください。それでも動かないときは先輩や上長に確認してください。

### タスクが見当たらないときの探し方

「あの案件、今どこにあるんだろう?」と探したいとき、以下の方法があります。

1. **検索機能を使う** — 画面上部の検索バーで入居者名や案件番号を入れる(最も早い)
2. **6.最新管理を開く** — 全案件が横断的に集まっているので、ここから探せる
3. **進行段階を推測してプロジェクトを順に開く** — 申込から日数が経っていそうなら 4.送金済み や 5.入居済み を確認

### ここまでのポイント

- タスクは「プロジェクト > セクション」の2階層で位置づけられる
- プロジェクト間移動はAsanaの自動化で行われる(手動操作は不要)
- 自分の作業完了 = 決められたフィールドの更新だけ
- フィールド更新順序を間違えるとトリガーが発火しないので、手順を守る
- タスクが見つからないときは検索バー、または6.最新管理を使う

## 2-5 「2.申込」での業務

### このプロジェクトに案件が来るまでの流れ

1. **韓国チーム** が入居者の希望条件を聞き取り、Asanaにタスクを作成
2. タスクは最初「2.申込」プロジェクトの「**申込準備中**」セクションに入る(この時点では仮申込)
3. **日本チームの別担当** が管理会社へ本申込を実行し、担当宅建士を割り振る
4. 本申込が完了すると、タスクは「**申込中**」セクションへ移動
5. 宅建士はここから業務を開始する

「申込準備中」セクションのタスクは宅建士の対応対象ではありません。「申込中」セクションのタスクから動き始めます。

### 宅建士が最初にやること(タスクを引き受けた直後)

**① タスクの担当者を自分に変更**

担当者欄が割り振り前の状態になっている場合、自分のアカウントを担当者に設定します。

**② 管理会社チャットのURLを説明欄に貼り付け**

管理会社とのやり取りを行うチャットツール(イタンジ申込受付くんなど)のURLを、タスクの説明欄に貼り付けます。これにより、後続工程の担当者も同じチャットにすぐアクセスできるようになります。

> 補足: 管理会社ごとに使用するチャットツールが異なります。

**③ 管理会社チャットの初回確認**

管理会社からの最初のメッセージを読み、案件の現状を把握します。

### 「2.申込」段階での主な業務

このプロジェクトでの主な業務は **管理会社との審査中のやり取り** と **審査通過までの進捗共有** です。

具体的には以下のようなやり取りが発生します。

- 管理会社からの追加書類依頼への対応(在留カードコピー、勤務先証明 等)
- 管理会社からの追加情報質問への回答
- 入居者側の事情変更の伝達
- 審査進捗の確認と社内共有

管理会社とのチャットを定期的に確認し、メッセージがあれば対応してください。社内への進捗共有は第3章「Slackでの社内共有」で扱います。

### 「2.申込」段階で宅建士が触らないフィールド

このプロジェクトでは多数のフィールドが表示されますが、以下は他部門の管轄です。

- **家電関連**(家電状況、家電内容、家電配達日 など)- 家電レンタル斡旋担当
- **lifeline** - 別担当
- **入金日** - 経理担当
- **노션이동(Notion移動)** - 韓国チーム

### 審査通過後の操作(プロジェクト移動)

審査結果が出たら、宅建士が以下のフィールドを更新します。この操作がトリガーとなり、Asanaの自動化ルールによってタスクの位置が変わります。

**① 起算日状況 を更新**

審査通過時点での起算日の決まり方によって、3つの値から選びます。

| 値 | 使う場面 |
|---|---|
| **決定** | 審査通過と同時に起算日が決定している場合 |
| **相談** | 審査は通過したが起算日はこれから決める(「○日から○日の間に決めてください」等の幅で提示された場合) |
| **未定** | 審査は通過したが前入居者の退去前のため、起算日の目安すら決まっていない場合 |

**② 審査結果 を更新**

| 値 | 使う場面 |
|---|---|
| **通過** | 審査通過した場合 |
| **落ち** | 審査落ちの場合 |
| **2番手** | 2番手として扱われている場合 |

**フィールド更新の順序**

> 重要: 「起算日状況」→「審査結果」の順で更新してください。順序を間違えると、Asanaの自動化ルールのトリガー条件を満たさず、タスクが「3.送金前」へ移動しない場合があります。

更新が完了すると、タスクは自動的に「3.送金前」プロジェクトへ移動します。これで「2.申込」での業務は一旦完了となり、続きは「3.送金前」プロジェクト側で行います。

### ここまでのポイント

- このプロジェクトでの業務は管理会社チャットでの審査中対応が中心
- 引き受け直後は担当者変更・URL貼り付け・チャット確認
- プロジェクト移動のトリガーは「起算日状況」→「審査結果」の順で更新

---

## 2-6 「3.送金前」での業務

### このプロジェクトに案件が来るタイミング

「2.申込」段階で「起算日状況」と「審査結果」を更新すると、タスクは自動的に「3.送金前」プロジェクトに移動します。このプロジェクトのセクション名は「**管理会社送金前**」です。

### 「3.送金前」段階での業務の全体像

このプロジェクトでは、以下の業務が並行して進みます。

1. 管理会社との **起算日調整**(必要な場合)
2. 入居者との **起算日の詳細詰め**(Slackで韓国チームに依頼)
3. 管理会社からの **初期費用請求書の受領と処理**
4. その後は経理担当が入金確認 → 管理会社送金 を実施

宅建士のキー操作は3番(初期費用請求書の処理)です。

### ① 管理会社との起算日調整

「2.申込」段階で「起算日状況 = 相談」または「未定」とした案件は、このプロジェクトに来た後も管理会社との調整が続きます。

- **相談** の場合: 管理会社から提示された期間内で起算日を確定させる
- **未定** の場合: 前入居者の退去予定が確定し次第、起算日の目安を管理会社と詰める

確定したら、タスクの「起算日」フィールドに日付を入力します。

### ② 入居者との起算日詰め

宅建士が直接入居者と連絡を取ることはなく、Slackで **韓国チーム** に依頼します(具体的なSlack操作は第3章で扱います)。

詰める内容: **起算日の希望日**

決まった起算日は「起算日」フィールドに入力します。

### ③ 初期費用請求書の処理(宅建士のキー操作)

これがこのプロジェクトで宅建士が行う最重要操作です。

**手順:**

| 順序 | 操作 | 補足 |
|---|---|---|
| 1 | 管理会社から送られてくる初期費用請求書(管理会社からUCへの請求書)をチャットで受領 | チャットツールは管理会社により異なる |
| 2 | 請求書PDFを **Googleドライブの「【申込チーム】請求書格納フォルダ」直下** にアップロード | サブフォルダには入れない |
| 3 | ファイル名を **「案件番号+入居者名+物件名.pdf」** に変更 | 例: `1729パクセイル(菱和パレス護国寺1004号室).pdf` |
| 4 | タスクの「**初期費用**」フィールドを「**作成依頼済**」に変更 | |

**補足:**
- 同フォルダ内に「テンプレート_契約金請求書.xlsx」がありますが、これは経理担当が使用するもので、宅建士は触りません
- 同フォルダ内の「請求書提出フォルダ」サブフォルダも、経理担当用のため宅建士は使いません

### ④ 経理担当による後続処理

宅建士が「作成依頼済」とした後、経理担当が以下を行います(宅建士の作業ではないが、流れの理解として知っておく)。

1. UCから入居者への請求書を作成 → 「作成済み」
2. 入居者へ請求書送付 → 「送付済み」
3. 入居者からUCへの入金確認 → 「入金済み」「入金日」入力
4. UCから管理会社へ送金 → 「管理へ送金 = 完了」

経理担当が「管理へ送金 = 完了」にした時点で、Asanaの自動化ルールによりタスクが「4.送金済み」プロジェクトへ移動します。

### 「初期費用」フィールドの担当マップ

| 値 | 担当 |
|---|---|
| 作成依頼済 | **宅建士** |
| 作成済み | 経理担当 |
| 送付済み | 経理担当 |
| 入金済み | 経理担当 |
| 確認済み | 経理担当 |

### 「3.送金前」段階で宅建士が触らないフィールド

- 入金日、管理へ送金 — 経理担当
- 家電関連 — 家電レンタル斡旋担当
- lifeline、電気・水道/ガス希望日 — 別担当
- 노션이동 — 韓国チーム

### ここまでのポイント

- 必ず行う操作は「**初期費用 = 作成依頼済**」への変更
- その前提として「**請求書PDFをGoogleドライブにアップロード**」を忘れない
- 起算日の調整は管理会社と韓国チームの両方を相手に進める
- 4.送金済みへの自動移動トリガーは経理担当の操作

---

## 2-7 「4.送金済み」での業務

### このプロジェクトに案件が来るタイミング

「3.送金前」段階で経理担当が「管理へ送金 = 完了」とすると、タスクは自動的に「4.送金済み」プロジェクトに移動します。

このプロジェクト内には複数のセクションがありますが、宅建士は特定のセクションを意識する必要はありません。タスクが「4.送金済み」プロジェクトに来ていれば、宅建士の対応対象です。

### 「4.送金済み」段階での宅建士業務

このプロジェクトには宅建士の中心業務である **IT重説の実施** が含まれます。Asana上の操作と、IT重説実務の流れを順を追って説明します。

### ① 契約書データの受領とGoogleドライブへの保管

管理会社から契約書類のデータ(雛形)が送られてきます。これは後工程で別チームが重説書類を作成するための元データです。

**手順:**

| 順序 | 操作 | 補足 |
|---|---|---|
| 1 | 管理会社からチャットなどで契約書データを受領 | 管理会社により受領方法は異なる |
| 2 | Asana「**契約書**」フィールドを **「雛形アップ済」** に変更 | この時点ではGoogleドライブにはまだ未アップロード |
| 3 | 受領したデータをGoogleドライブ「**【申込チーム】重要事項作成フォルダ**」直下にアップロード | サブフォルダには入れない |
| 4 | Asana「契約書」フィールドを **「作成依頼済」** に変更 | アップロード完了の合図 |

> なぜ2段階に分けるか: 「雛形アップ済」は受領済みだがドライブ未アップ、「作成依頼済」はドライブアップ完了を示します。別チームが「重説作成に取り掛かれる状態かどうか」を判別するため、この段階的なステータス管理を行っています。

### ② 別チームによる重説書類作成(宅建士の作業ではない)

「作成依頼済」となった後、別チームが重説書類を作成します。「契約書」フィールドは別チームによって以下のように段階更新されます:

- 作成済み → 修正中 → 重説作成済

宅建士はこの間、別の案件を進めながら状態を見守ります。

### ③ 重説書類の事前確認

「重説作成済」または「契約書送信済」になった後、宅建士は重説書類の内容を事前にチェックします。

> 詳細は **第4章「重説前の書類確認」** で扱います。

### ④ IT重説の実施

事前確認が完了したら、入居者に対してIT重説を実施します。

> 詳細は **第5章「IT重説の実施」** で扱います。

### ⑤ IT重説後の処理

IT重説終了後、別担当が Asana「契約書」フィールドを **「読み実施済み」** に更新します。これは宅建士の作業ではありません。

### 在留カードの管理会社への送付(必要時)

管理会社によっては、契約後に入居者の在留カードのコピーやデータ提出を求めてくる場合があります。

- Slackで韓国チームに在留カードデータの取得を依頼
- 受け取ったデータを管理会社チャット経由で送付

すべての管理会社で必要なわけではなく、要求のある会社のみ対応します。

### 「契約書」フィールドの担当マップ

| 値 | 担当 |
|---|---|
| **雛形アップ済** | **宅建士** |

## 3-1 Slackチャンネルの全体像

UCJapanでは、業務の社内連絡を **Slack** で行っています。多数のチャンネルがありますが、宅建士が日常的に使うチャンネルは以下の4つです。

| # | チャンネル名 | 用途 |
|---|---|---|
| 1 | **契約チーム** | 契約・重説まわりの社内連絡 |
| 2 | **質問_韓国チームへ** | 日本側から韓国チームへの依頼・質問 |

このほかにもチャンネルは多数ありますが、いずれも他部門が主に使うもので、宅建士が能動的に投稿することはありません。サイドバーに表示されていても、これら4つ以外は普段気にする必要はありません。

### 4チャンネルの役割の違い

**契約チーム(日本チーム内専用)**

**質問_韓国チームへ(日本→韓国)**
- 主に日本チームから韓国チームへの依頼を投稿
- 入居者ヒアリングが必要な内容、書類取得依頼、確認依頼など
- 入居者と直接コミュニケーションを取るのは韓国チーム

**質問_日本チームへ(韓国→日本)**
- 主に韓国チームから日本チームへの確認や依頼が投稿される
- 宅建士は基本的に **受け手** として読み、自分宛のメンションに返答
- 入金関連、書類関連、入居者の意向伝達など

**asana_日本チーム(自動通知)**
- Asanaからの自動通知が流れる
- 宅建士は **読むだけ**(返信不要)
- 重要な通知(起算日決定、読み合わせ決定、担当者振り当て、申込キャンセル等)を見逃さないようにする

### 共通の投稿フォーマット

「契約チーム」「質問_韓国チームへ」「質問_日本チームへ」の3チャンネル(=人間が手動で投稿するチャンネル)では、共通して以下の投稿フォーマットが使われています。

```
@担当者メンション(必要に応じて)
[案件番号][入居者名(物件名 部屋番号)]
本文(質問・依頼・連絡内容)
```

**例:**
```

契約日決まってますか?
```

### ここまでのポイント

- 宅建士が日常的に使うのは4チャンネル
- それぞれ役割が異なる(発信用・依頼用・受信用・自動通知用)

## 3-5 asana_日本チーム

### このチャンネルの位置づけ

「asana_日本チーム」は、**Asanaからの自動通知** が流れるチャンネルです。手動投稿はほとんどなく、Asanaアプリが特定のイベント発生時にメッセージを自動投稿します。

宅建士はここで **読むだけ** が基本です。返信や行動は、通知から該当タスクへ飛んでAsana上で行います。

### 主な自動通知の種類

**①【振り当て】担当者の割り当て通知**

タスクに担当者が割り当てられたとき、自動で通知されます。

```
【振り当て】
担当者振り当て済み
🟦 リサ
1745ベソヨン(メインステージ武蔵関 806)
배소영님 작성해주신 내용 확인 부탁드리겠습니다.
[월세] - 〜¥85,000 (도쿄 원룸 평균)
```

新しい案件が割り振られたときの最初の通知。これを見て、自分の担当案件を把握します。

**②【起算日決定】起算日確定の通知**

起算日が確定したとき、自動で通知されます。

```
【起算日決定】
お客様が起算日を選びました。確認お願いします。
🟦 ucjapanllc@gmail.com
1713ハジョンミン(スカイコート後楽園第5 405)
起算日「5/25〜5/28」で決めてください!
```

通知を見て、起算日に対する次の動き(管理会社への伝達、関連スケジュールの確認など)を進めます。

**③【○○読み合わせ決定】IT重説日程の確定通知**

IT重説の日程が確定したときの通知。タイトルの「○○」部分は管理会社により変わります(例: 【CIC読み合わせ決定】など)。

```
【CIC読み合わせ決定】
日にちを確認してください
Irei Daisuke
1706キムミンジ(ルーブル練馬高野台弐番館 303)
起算日5/28...
```

自分のカレンダーと照合し、IT重説の準備を始めます。

**④【申込キャンセル】申込キャンセル通知**

申込がキャンセルになったときの通知。

```
【申込キャンセル】
説明のところで理由を確認してください。
1738ファンイェナ(スカイコート浅草第3 302)
황예나님 작성해주신 내용 확인 부탁드리겠습니다.
```

自分の担当案件がキャンセルされた場合、止めるべき作業(管理会社への送金、書類作成依頼など)がないか確認し、必要に応じて関係者へ連絡します。

### 通知に含まれる情報

各通知には自動的に以下が含まれます:

- 通知タイトル(【○○】)
- 簡単な説明文
- 担当者情報
- 該当タスクのリンク(タスク名: 案件番号+入居者名+物件名)
- 案件詳細の一部
- 「**Asanaで作業を見る**」ボタン

「Asanaで作業を見る」をクリックすると、ブラウザでAsanaの該当タスクに直接飛べます。

### 宅建士の確認方法

このチャンネルは1日1回以上は目を通します。特に以下のタイミングで確認すると漏れが少なくなります:

- **朝の業務開始時** — 前日夜から朝までに来た通知を確認
- **担当案件の起算日前後** — 起算日決定通知を見逃すと後続工程が遅れる
- **IT重説実施日の前日** — 読み合わせ決定通知の最終確認

### ここまでのポイント

- asana_日本チームは **Asanaからの自動通知専用** チャンネル
- 主な通知は【振り当て】【起算日決定】【○○読み合わせ決定】【申込キャンセル】の4種類
- 宅建士は **読むだけ** が基本、対応はAsanaタスク上で行う
- 1日1回以上は目を通して、自分宛の通知を見逃さないようにする

# 第4章 重説前の書類確認

## 4-1 重説書類のチェックを始めるタイミング

重説書類は宅建士が作成するのではなく、別チーム(重説作成担当)が作成します。宅建士は完成した重説書類の内容を **事前にチェック** する役割です。

### チェック開始のタイミング

Asanaタスクの「契約書」フィールドが以下の値になったら、重説書類のチェックを始められます:

- **重説作成済** — 別チームが重説書類を作成完了した状態
- **契約書送信済** — 管理会社への送信前後の段階

これより前の段階(雛形アップ済、作成依頼済、作成済み、修正中)では、まだ重説書類がチェック可能な状態ではないため、チェック作業は不要です。

### チェックの所要時間

簡易チェックなので、1案件あたり数分〜10分程度が目安です。重説実施までに余裕を持って完了させます。

### ここまでのポイント

- 重説書類は別チームが作成、宅建士はチェック担当
- チェック開始は「契約書」フィールドが「重説作成済」または「契約書送信済」になってから
- 簡易チェックで完結する作業

---

## 4-2 重説書類とは

### 重説書類の位置づけ

重説書類(重要事項説明書)は、宅地建物取引業法第35条で定められた、賃貸借契約の前に宅地建物取引士が入居者に対して説明しなければならない書類です。

UCJapanの業務フローでは、以下のような書類群を扱います:

| 書類 | 概要 | 作成元 |
|---|---|---|
| **契約書類雛形** | 管理会社が発行する元データ | 管理会社 |
| **重要事項説明書(重説)** | 入居者に説明する公式書類 | 別チーム(契約書類雛形をもとに作成) |
| **賃貸借契約書** | 入居者と締結する契約書 | 別チーム(管理会社雛形をもとに作成) |

### 書類間の関係

- 管理会社が **契約書類雛形** を発行する(元データ)
- 別チームがこの雛形を参照しながら、入居者向けの **重説書類** を作成する
- 宅建士は完成した重説書類を、管理会社雛形と照合してチェックする

### 宅建士がチェックする書類

事前確認の対象は、別チームが作成した **重説書類** です。管理会社の雛形そのものをチェックするわけではありません。

ただし、チェック時には管理会社の雛形と重説書類を **照合** することで、転記ミスや数字の誤りを発見します。

### ここまでのポイント

- UCJapanでは「管理会社雛形」→「重説書類(別チーム作成)」という流れで書類が作られる
- チェック対象は別チーム作成の **重説書類**
- 雛形と重説書類を照合してチェックする

---

## 4-3 主要チェック項目

### チェックの基本姿勢

重説書類のチェックは、**管理会社雛形と重説書類を照合して、転記ミスや数字の誤りがないかを確認する** 作業です。法的妥当性の判断や、契約条件そのものの是非を問う作業ではありません。

特に **数字や固有名詞の誤り** は重説実施時に問題となるため、これらを重点的にチェックします。

### 主要チェック項目

| カテゴリ | 項目 | チェックポイント |
|---|---|---|
| **物件情報** | 物件名 | 管理会社雛形と一致しているか(誤字、表記揺れ) |
| | 部屋番号 | 数字が正確か(101と102のような近似ミスに注意) |
| | 所在地 | 住所が一致しているか |
| **契約条件** | 契約開始日(起算日) | Asanaタスクの「起算日」フィールドと一致しているか |
| | 契約期間 | 開始日と終了日の整合 |
| | 月家賃 | 金額が雛形と一致しているか |
| | 敷金 | 月数や金額に誤りはないか |
| | 礼金 | 月数や金額に誤りはないか |
| | その他の費用 | 共益費、管理費、保険料などの数字 |
| **入居者情報** | 入居者名(カタカナ表記) | スペルミス、表記揺れがないか |
| | 連絡先 | 携帯番号、メールアドレスの誤り |
| **その他** | 管理会社名 | 正確な社名表記 |
| | 保証会社名 | 正確な社名表記 |

### チェックの進め方

1. 重説書類を開き、管理会社雛形と並べる
2. 上記の項目を上から順に照合する
3. 数字や固有名詞に誤りがないか目視確認
4. 不備があれば箇所を控えておく(差し戻し対応は4-4で扱う)

### 特に注意すべきポイント

**a) 数字の桁数**
家賃や敷礼金は1桁の誤りで大きな金額差が生じます。「85,000」と「850,000」のような桁ずれに注意。

**b) 日付の年月日**
特に契約開始日(起算日)は、Asanaタスク・管理会社雛形・重説書類の3つで一致している必要があります。

**c) 入居者名のカタカナ表記**
韓国名のカタカナ転写は表記揺れが起きやすい部分です(「キム」/「金」、「ジュンヒョン」/「ジョンヒョン」など)。雛形と完全一致しているかを確認します。

### ここまでのポイント

- 主要チェックは **物件名・部屋番号・契約開始日・金額・入居者名** など
- 雛形との照合作業 — 法的判断は不要
- 数字の桁ずれ、日付の年月日、カタカナ表記の揺れに特に注意

---

## 4-4 不備発見時の対応

### 不備発見時の動き

チェック時に重説書類に不備を見つけたら、**Slack「契約チーム」で連絡** します。

投稿のフォーマットは第3章で扱った通り:

```
@(担当者)
[案件番号][入居者名(物件名 部屋番号)]
不備の内容(どこに何の誤りがあるか)
```

連絡を受けた別チームが、修正対応とAsana「契約書」フィールドの更新(「差し戻し」値への変更など)を行います。

### Asana「契約書」フィールドの「差し戻し」について

「差し戻し」は別チームが操作する値です。宅建士が直接「差し戻し」に変更することはありません。

### ここまでのポイント

- 不備を見つけたら **Slack「契約チーム」** で連絡
- Asana「差し戻し」フィールドは別チームが操作 — 宅建士は触らない

# 第5章 IT重説の実施

## 5-1 IT重説の概要と法的要件

### IT重説とは

IT重説とは、ビデオ通話システムを利用して行う重要事項説明のことです。従来の対面方式に代わり、入居者と宅建士が遠隔地にいる状態で重要事項説明を実施できる方式として、2017年から賃貸取引で本格運用が始まりました。

UCJapanでは、入居者が海外(主に韓国)から渡日する前に契約を完了させる必要があるため、IT重説が標準的な実施方式となっています。

### 法的要件

IT重説を有効に実施するには、宅建業法に基づく以下の要件を満たす必要があります。

| 要件 | 内容 |
|---|---|
| **双方向通信** | 宅建士と入居者がリアルタイムで映像と音声をやり取りできる環境であること |
| **宅建士証の提示** | 宅建士が画面上で宅建士証を提示し、入居者が文字を判読できる状態であること |
| **事前の書類交付** | 入居者が重要事項説明書を実施前に受け取っており、説明中に手元で確認できる状態であること |
| **入居者の意思確認** | 入居者がIT重説に同意していること、説明を受ける環境が整っていることを確認すること |

これらが満たされていない場合、IT重説は無効となります。実施前に必ず確認します。

### UCJapanでの実施方式

| 項目 | 内容 |
|---|---|
| **使用ツール** | Google Meet |
| **実施者** | 宅建士(UCJapan所属) |
| **日程確定の経路** | Asana自動通知「【○○読み合わせ決定】」がSlack「asana_日本チーム」に流れる、もしくは Slack「質問_日本チームへ」で韓国チームから日程提案 |
| **重説書類の事前送付** | IT重説実施前に、入居者へ重説書類を送付済みであること(別チームが対応) |

### ここまでのポイント

- IT重説は法的要件を満たして実施する必要がある(双方向通信、宅建士証提示、事前書類交付、本人同意)
- UCJapanでは **Google Meet** を使用
- 日程確定は Asana自動通知 または Slack経由で把握

---

## 5-2 IT重説の事前準備

IT重説を円滑かつ法的に有効に実施するため、実施前に以下の準備を整えます。

### 書類の準備

| 書類 | 確認ポイント |
|---|---|
| **重要事項説明書(重説書類)** | チェック済みのもの(第4章参照)。入居者にも事前送付済みであることを確認 |
| **賃貸借契約書** | 重説と内容に齟齬がないか |
| **宅建士証** | 画面で提示できる状態 |

### 機材・環境の準備

| 項目 | 確認ポイント |
|---|---|
| **PC・カメラ・マイク** | 映像と音声が正常に出力されること |
| **通信環境** | 安定したネット回線 |
| **静かな場所** | 雑音や第三者の声が入らない場所 |

### Google MeetのURL

Google MeetのURLは **韓国チームが発行** し、入居者と宅建士の双方に共有します。宅建士はURL発行を行いません。

実施前にURLが共有されていることを確認し、事前にアクセステストをしておくと安心です。

### 入居者側の準備(韓国チーム経由で確認)

入居者側の準備は韓国チームが確認・サポートします。宅建士は以下の項目について、韓国チームから「準備完了」の連絡が来ていることを確認します。

| 項目 |
|---|
| 重説書類の受領 |
| 本人確認書類(パスポート等)の準備 |
| 通信環境の確保 |
| IT重説への同意 |

### 録画について

IT重説の録画は **ケースによる** ため、案件ごとに判断します。録画する場合は、開始時に入居者の同意を得てから録画を開始します。

### ここまでのポイント

- 書類・機材・通信環境を事前に整える
- Google MeetのURLは **韓国チームが発行・共有**
- 入居者側の準備は韓国チームが対応するので、宅建士は完了連絡を確認
- 録画はケースバイケース

---

## 5-3 IT重説の進行手順

### 言語と進行体制

UCJapanのIT重説は、以下の体制で進行します。

- **宅建士が日本語で説明**(法的な説明責任は宅建士が負う)
- **韓国チームが同席し、必要に応じて韓国語で補足**(入居者の理解を確実にする)

入居者が日本語のみで理解できない場合、韓国チームが補足説明や通訳的な役割を果たします。

### 全体の流れ

```
1. 開始・接続確認
       ↓
2. 本人確認
       ↓
3. 宅建士証の提示
       ↓
4. IT重説実施の意思確認
       ↓
5. 重説書類の説明(重要項目)
       ↓
6. 質疑応答
       ↓
7. 終了
```

### 1. 開始・接続確認

Google Meetに入室し、入居者と韓国チームメンバーの入室を待ちます。全員揃ったら、以下を確認します。

- 映像が双方向で見えているか
- 音声が双方向で聞こえているか
- 入居者が静かな場所にいて、説明を受けられる状況か

### 2. 本人確認

入居者にパスポート等の身分証明書を画面に映してもらい、本人であることを確認します。

確認する内容:
- 氏名
- 顔写真と画面上の入居者本人が一致しているか

### 3. 宅建士証の提示

宅建士証を画面に映し、入居者が文字を判読できる状態で提示します。

- 宅建士証の文字が読めるか入居者に確認
- 宅建士の氏名・登録番号を伝える

これはIT重説の法的要件の一つです。**省略しないこと**。

### 4. IT重説実施の意思確認

入居者に対して、これからIT重説を行うことについて改めて同意を確認します。

例:
> 「これから重要事項説明をIT重説として実施しますが、よろしいでしょうか?」

### 5. 重説書類の説明(重要項目)

重説書類は **全項目を逐語で読み上げる必要はなく、重要項目を中心に要点を説明** します。

説明の進め方:
- 入居者に書類のページを指示する(「2ページ目をご覧ください」など)
- 該当箇所の要点を日本語で説明
- 韓国チームが必要に応じて韓国語で補足
- 重要なポイントは入居者に理解確認を取る

### 6. 質疑応答

説明が一通り終わったら、入居者からの質問を受け付けます。質問がなければ「不明な点はありませんか?」と明示的に確認します。

韓国チームがいるので、入居者が日本語で質問しにくい場合は韓国語で質問してもらい、宅建士に韓国チームが伝える形でも問題ありません。

### 7. 終了

入居者の質問が無くなったら、重説終了を宣言します。

### ここまでのポイント

- 宅建士が日本語で説明、韓国チームが補足
- 重要項目を中心に要点説明(全項読み上げ不要)
- 本人確認と宅建士証提示は法的要件 — 省略不可

---

## 5-4 IT重説後の処理

IT重説が完了したら、以下の処理を行います。

### 1. 社内共有について

IT重説の完了情報は、同席している韓国チームによってリアルタイムに社内共有されます。宅建士が改めて社内連絡を行う必要はありません。

### 2. Asana「契約書」フィールドの更新は別担当

IT重説完了後、Asana「契約書」フィールドが **「読み実施済み」** に更新されますが、これは **別担当** が行います。宅建士自身が「読み実施済み」に変更する必要はありません。

### 3. 入居者の電子契約への移行

IT重説完了後、入居者は電子契約(GMOサインなど)に進みます。電子契約の手続きは入居者自身が行い、宅建士は基本的には関与しません。

電子契約完了の連絡が韓国チームから「質問_日本チームへ」に来ることがあります。例:
```
@(宅建士)
[案件番号][入居者名(物件名 部屋番号)]
GMOサイン終わりました。
```

### 4. 後続工程の見守り

IT重説完了後、案件は以下の流れで進みます(宅建士の直接の業務範囲外):

- 電子契約完了 → 別チームが「契約書」フィールドを進める
- 鍵渡し → 別担当が対応
- 入居完了 → タスクが「5.入居済み」プロジェクトへ移動

宅建士は、何か追加対応が必要になった場合(管理会社からの追加書類依頼、入居者連絡先の伝達など)に対応します。

### ここまでのポイント

- 社内共有は韓国チームがリアルタイム対応
- Asana「読み実施済み」は **別担当が更新** — 宅建士は触らない
- 入居者の電子契約は宅建士の業務範囲外
- 後続工程は他部門が進める、宅建士は追加対応のみ

---

## 5-5 トラブル対応

IT重説中にトラブルが発生した場合は、原則として **IT重説を中断し、別日に再設定** します。法的要件を満たせない状態でIT重説を続けても無効になるため、無理に続行しないことが大切です。

### よくあるトラブル

| トラブル | 主な対応 |
|---|---|
| 通信が切れる・不安定 | 再接続を試みる。安定しなければ中断して別日再設定 |
| 入居者が身分証明書を準備していない | 中断して別日再設定 |
| 入居者の手元に重説書類がない | 中断して別日再設定 |
| 入居者が中断を希望した | 中断して別日再設定 |

### 再設定について

再設定の日程調整は韓国チームを通じて入居者と再度行います。詳細は通常のIT重説実施フローと同じです(5-1〜5-2参照)。

### ここまでのポイント

- トラブル時は無理に続行せず中断する
- 中断後の再設定は韓国チーム経由
